Метки и управление потоком
==========================

.. _label-statement:

Инструкция Label
----------------

Инструкции меток (label) позволяют присвоить заданное имя определённой точке в программе.
Они существуют исключительно для того, чтобы к ним можно было перейти с помощью вызова (call) или перехода (jump) из скрипта Ren'Py, функций Python или экранов. ::

    label sample1:
        "Здесь метка 'sample1'."

    label sample2(a="default"):
        "Здесь метка 'sample2'."
        "a = [a]"

Инструкция метки может иметь связанный с ней блок кода. В этом случае управление
переходит в блок при достижении инструкции метки и продолжается с инструкции,
следующей за меткой, когда достигается конец блока: следующий код при переходе к метке "origin"
выведет последовательность "a, b, c". ::

    label origin:
    "a"
    label hasblock:
        "b"
    "c"
    return

Существует два вида меток: *глобальные* и *локальные*. Глобальные метки
находятся в одном общем глобальном пространстве имён для всех файлов проекта и,
следовательно, должны иметь уникальные имена в пределах игры. Локальная метка,
в свою очередь, относится к глобальной метке, поэтому несколько локальных меток
в игре могут иметь одинаковое имя, при условии, что они относятся к разным
глобальным меткам. Чтобы объявить локальную метку, добавьте к её имени префикс
в виде точки ``.`` и разместите её под глобальной меткой, к которой она будет
принадлежать.
Например::

    label global_label:
        "Под глобальной меткой..."
    label .local_label:
        "...располагается локальная."
        jump .another_local
    label .another_local:
        "И ещё одна!"
        jump .local_label

Локальную метку также можно объявить по её полному имени, состоящему из имени
глобальной метки и имени локальной метки, разделённых точкой. При этом
глобальная метка не будет объявлена. ::

    label global_label.yet_another_local:
        "И ещё одна!"
        jump .local_label

На локальные метки можно ссылаться напрямую внутри той же глобальной метки,
где они объявлены, или по их полному имени, состоящему из глобальной и локальной
частей::

    label another_global:
        "Теперь перейдём к локальной метке, расположенной где-то ещё."
        jump global_label.local_name

Инструкция метки может принимать необязательный список параметров. Эти параметры
обрабатываются так, как описано в :pep:`570`, за двумя исключениями:

Во-первых, значения параметров по умолчанию вычисляются во время вызова.

Во-вторых, переменные имеют динамическую, а не лексическую область видимости.
Это означает, что когда переменная получает свое значение из параметра метки,
её значение будет возвращено к предыдущему состоянию (к предыдущему значению,
если оно было, или к отсутствию переменной в противном случае) при достижении
инструкции return. Это также означает, что в зависимости от того, как была
достигнута инструкция, использующая определённую переменную, эта переменная может
получить своё значение из параметра метки, а может и не получить; такое
поведение невозможно в чистом коде Python. ::

    default a = 3

    label start:
        menu:
            "Call":
                call label_with_params(5)
            "Jump":
                jump label_without_params
        jump start

    label label_with_params(a):
    label label_without_params:
        e "a = [a]" # отображает 5 или 3 в зависимости от выбранного пути
        return

Обычно не имеет смысла переходить к метке с параметрами с помощью jump или
предыдущей инструкции. Пример меток с параметрами см. в разделе об
:ref:`инструкции call <call-statement>`.


.. _jump-statement:

Инструкция Jump
---------------

Инструкция jump используется для передачи управления указанной метке.

Если присутствует ключевое слово ``expression``, следующее за ним выражение
вычисляется, и полученная в результате строка используется как имя метки для
перехода. Если ключевое слово ``expression`` отсутствует, имя метки для перехода
должно быть указано явно.

Можно передать имя локальной метки, как с ``expression``, так и без, либо
с префиксом в виде имени глобальной метки ("global_label.local_label"), либо
с точкой в начале (".local_label").

В отличие от call, jump не помещает следующую инструкцию в стек вызовов.
В результате нет способа вернуться в то место, откуда был совершён переход. ::

    label loop_start:

        e "О нет! Кажется, мы попали в бесконечный цикл."

        jump loop_start

.. _call-statement:

Инструкция Call
---------------

Инструкция call используется для передачи управления указанной метке. Она
также помещает следующую инструкцию в стек вызовов, позволяя инструкции return
вернуть управление инструкции, следующей за call.

Если присутствует ключевое слово ``expression``, следующее за ним выражение
вычисляется, и полученная в результате строка используется как имя метки для
вызова. Если ключевое слово ``expression`` отсутствует, имя метки для вызова
должно быть указано явно.

Можно передать имя локальной метки, как с ``expression``, так и без, либо
с префиксом в виде имени глобальной метки ("global_label.local_label"), либо
с точкой в начале (".local_label").

Если присутствует необязательное выражение ``from``, оно фактически добавляет
инструкцию метки с заданным именем сразу после инструкции call. Явная метка
помогает гарантировать, что сохранённые игры со стеками возврата смогут
вернуться в нужное место при загрузке с изменённым скриптом.

Инструкция call может принимать аргументы, которые обрабатываются так, как
описано в :pep:`448`.

При использовании выражения call со списком аргументов необходимо вставить
ключевое слово ``pass`` между выражением и списком аргументов. В противном
случае список аргументов будет разобран как часть выражения, а не как часть
вызова. ::

    label start:

        e "Сначала мы вызовем подпрограмму."

        call subroutine

        call subroutine(2)

        call expression "sub" + "routine" pass (count=3)

        return

    # ...

    label subroutine(count=1):

        e "Я пришёл сюда [count] раз(а)."
        e "Далее мы вернёмся из подпрограммы."

        return

.. warning::

    Выпускать игру без выражений ``from`` для каждой инструкции ``call``
    опасно, если вы планируете выпускать обновления для игры в будущем.
    Если такие выражения не добавлены и вы редактируете файл, содержащий
    инструкцию ``call``, существует потенциальный риск того, что сохранения,
    сделанные внутри вызванной метки, могут повредиться.

    Использование опции "Add from clauses to calls" при сборке дистрибутива
    игры может решить эту проблему.

.. _return-statement:

Инструкция Return
-----------------

Инструкция ``return`` извлекает верхнюю инструкцию из стека вызовов и передаёт
ей управление. Если стек вызовов пуст, инструкция return перезапускает Ren'Py,
возвращая управление в главное меню.

Если инструкции return передаётся необязательное выражение, оно вычисляется, и
его результат сохраняется в переменной ``_return``. Эта переменная имеет
динамическую область видимости для каждого контекста.

.. _special-labels:

Специальные метки
-----------------

Следующие метки используются Ren'Py:

``start``
    По умолчанию Ren'Py переходит к этой метке при запуске игры.

``quit``
    Если эта метка существует, она вызывается в новом контексте, когда пользователь
    выходит из игры.

``after_load``
    Если эта метка существует, она вызывается при загрузке игры. Её можно
    использовать для исправления данных при обновлении игры. Если данные
    изменяются этой меткой, следует вызвать :func:`renpy.block_rollback`,
    чтобы предотвратить отмену этих изменений, если игрок откатится за точку
    загрузки.

``before_load``
    Если эта метка существует, она вызывается в новом контексте непосредственно
    перед загрузкой игры. Её можно использовать для подготовки к загрузке,
    например, для смены музыки или отображения анимации.

    Как только эта метка вернёт управление, Ren'Py продолжит загрузку. Экран
    не будет обновляться до завершения загрузки, поэтому анимации не будут
    воспроизводиться. Когда загрузка завершится, состояние игры будет заменено
    загруженным состоянием, и поэтому изменения, сделанные в этой метке,
    будут утеряны.

``splashscreen``
    Если эта метка существует, она вызывается при первом запуске игры, перед
    показом главного меню. Пожалуйста, обратитесь к разделу
    :ref:`Добавление заставки <adding-a-splashscreen>`.

``before_main_menu``
    Если эта метка существует, она вызывается перед главным меню. Она используется
    в редких случаях для настройки главного меню, например, для запуска видео
    на заднем плане.

``main_menu``
    Если эта метка существует, она вызывается вместо главного меню. Если она
    возвращает управление, Ren'Py начнёт игру с метки ``start``. Например,
    следующий код немедленно начнёт игру, не показывая главное меню. ::

        label main_menu:
            return

``after_warp``
    Если эта метка существует, она вызывается после warp, но до выполнения
    инструкции, к которой был совершён переход. Пожалуйста, обратитесь к разделу
    :ref:`Переход к строке <warping_to_a_line>`.

``hide_windows``
    Если эта метка существует, она вызывается, когда игрок скрывает окна
    правой кнопкой мыши или клавишей H. Если она возвращает true, скрытие
    отменяется (предполагается, что скрытие уже произошло). В противном случае
    скрытие продолжается.

Ren'Py также использует следующие метки для отображения некоторых из :doc:`специальных экранов <screen_special>`:

* ``main_menu_screen``
* ``load_screen``
* ``save_screen``
* ``preferences_screen``
* ``joystick_preferences_screen``

Контексты
---------

Контексты используются внутри Ren'Py для управления изменяемым и сохраняемым
состоянием игры. Контексты включают:

* текущую выполняемую инструкцию Ren'Py,
* стек вызовов, как описано выше, а также имена и предыдущие значения
  динамических переменных, созданных :func:`renpy.dynamic`,
* изображения, отображаемые в данный момент (и информация о них, такая как их
  атрибуты, применённые к ним трансформации и т. д.),
* отображаемые экраны и переменные внутри них,
* аудио, которое воспроизводится или находится в очереди.

Большую часть времени используется только один контекст, и существует только один
экземпляр каждого из этих элементов. Ситуация меняется при входе в главное или
игровое меню; всё перечисленное выше может быть изменено и будет восстановлено
при выходе из контекста меню. Некоторые из этих изменений происходят
автоматически, например, слой экранов очищается при входе в контекст.

Ren'Py также создаёт новые контексты в рамках :ref:`повтора (replay) <replay>`
и при :func:`скрытии интерфейса <HideInterface>`.

Создание :ref:`языка экранов <screens>` значительно уменьшило потребность
в создании контекстов.

Откат (Rollback) включён только в базовом контексте (то есть, когда существует
только один контекст), и только базовый контекст сохраняется, поэтому игровое
меню использует контекст.
